Introduction

When working with OpenCV functions that utilize CUDA, it’s important to be aware of the initialization delay that occurs the first time you call such functions. This blog post aims to shed light on what is meant by an initialization delay and the two main factors contributing to it. Understanding this delay and its causes should allow you to more effectively manage the initialization process. In this blog, we will explore the creation of a CUDA context and the JIT compilation of PTX code as the primary causes of this delay.

The Initialization Delay:

The initialization delay is a brief period that occurs during the first invocation of any OpenCV function that employs CUDA. It arises due to two main factors:

  1. Creation of a CUDA context: The first cause of initialization delay arises from the creation of a CUDA context which is analagous to a CPU process. When calling an OpenCV function utilizing CUDA, a CUDA context is created, which always incurs a certain delay.

  2. Just-In-Time compilation. Depending on your build configuration and the hardware you are executing CUDA functions on, PTX (Parallel Thread Execution) code can be compiled into device-specific binary code. This compilation happens during the first invocation and adds to the initialization delay.

Understanding the Delay:

It’s essential to acknowledge the existence of the initialization delay, especially when timing your code. Incorrectly attributing this delay to the actual computation can lead to inaccurate performance measurements. Furthermore, understanding the source of the delay enables you to optimize your code effectively.

Managing the Delay:

Unlike the delay from JIT compilation (cause #2) the delay from CUDA context creation (cause #1) cannot be removed It is possible to completely remove the delay from JIT compilation (cause #2) but not from CUDA context creation (cause #1). That said by creating the CUDA context in advance or performing context setup during non-time-sensitive stages of your program, you can eliminate the delay from subsequent function calls.

Demonstrating the Initialization Delay:

To demonstrate the initialization delay, we will utilize two simple functions: time_set_device() and time_abs() shown below. These functions showcase the delay by measuring either

  • the time required to call cv.cuda.setDevice() a function which doesn’t perform any computations but is still subject to both types of initialization delay.
  • the time required to perform a simple calculation using CUDA.
from timeit import default_timer as timer
import cv2 as cv

def time_set_device():
    start = timer()
    cv.cuda.setDevice(0)
    end = timer()
    print(f'Initialization time: {end-start:.2f}s')
    
def time_abs():
    start = timer()
    src_dst = cv.cuda.GpuMat(1000,1000,cv.CV_8U,100)
    cv.cuda.abs(src_dst,src_dst)
    end = timer()
    print(f'Initialization time: {end-start:.4f}s')

Just-in-Time (JIT) compilation

For more information on JIT compilation see Choosting a suitable CUDA compute capability

Under normal circumstances, when no additional CUDA Environmental Variables have been set, OpenCV handles the JIT compilation process automatically. If an OpenCV CUDA function is called and the library does not have pre-built binary code suitable for your device, but there is PTX code available with a compute capability less than or equal to that required by your device, the following steps occur:

  1. JIT Compilation to Binary Code: The PTX code is dynamically compiled (JIT compiled) into device-specific binary code. This compilation process incurs an initial delay.

  2. Caching of Binary Code: The resulting binary code is then cached in a default location and stored in the system’s RAM. This caching ensures that the binary code is readily available for subsequent program executions.

This compilation causes an initial delay, but because the JIT compiled binary code is stored in RAM there will be no additional delay during the lifetime of the program for any functions which are part of the same OpenCV module as the one the function you called is part of. Additionaly as long as the cached entry is not deleted there will be no delay when running the same program again.

example

Let’s consider a scenario where your main GPU is an RTX 2080 with a compute capability of 7.5, and you own additional GPUs with compute capabilities starting from 6.1 (e.g., GeForce 10 series and newer). Under these circumstances its perfectly reasonable to have built OpenCV with the following CMake flags

-DCUDA_ARCH_PTX=6.1 -DCUDA_ARCH_BIN=7.5

Now, let’s say you attempt to execute CUDA functions in OpenCV on an RTX 3070, which has a compute capability of 8.6. In this situation, you will encounter a delay as the PTX code initially compiled for compute capability 6.1 needs to be JIT compiled into binary code compatible with the RTX 3070’s compute capability of 8.6.

To demonstrate the concept, I will execute a single CUDA function from Python, which is linked against OpenCV. In this demonstration, I have intentionally used the opencv_worldxxx module to showcase the maximum delay. By using this module, JIT compilation of all OpenCV modules that utilize CUDA will be triggered.

To simulate the scenario on an RTX 3070, where OpenCV was built with the flags -DCUDA_ARCH_PTX=8.6 -DCUDA_ARCH_BIN=8.6, we can take the following steps. Under normal circumstances, this configuration would not require JIT compilation. However, to avoid the need to recompile OpenCV, we can simulate OpenCV being built with only the -DCUDA_ARCH_BIN=8.6 flag.

To achieve this simulation, we can set the environmental variable CUDA_FORCE_PTX_JIT to 1 before calling any CUDA functions. This configuration ensures that the PTX code is forced to undergo JIT compilation, simulating the absence of pre-compiled binary code specifically tailored for the RTX 3070.

By executing this CUDA function while emulating the mentioned configuration, we can observe the resulting delay caused by the JIT compilation process. This demonstration provides a practical example of how the presence or absence of pre-compiled binary code impacts the initialization delay when utilizing CUDA functions in OpenCV.

Initialization time: 0.00s

there isn’t any delay because the JIT compiled binary code has been stored in RAM and the CUDA context created. Additionally because we are using the world module (opencv_worldxxx) all of the CUDA code has been JIT compiled, so calling time_abs() as below

time_abs()
Initialization time: 0.0013s

If we had not compiled the world module then JIT is applied to each module seperately. e.g. Calling cv.cuda.set_device() would only JIT compile the CUDA functions which are part of opencv_corexxx and we would therefore get a delay when calling cv.cuda.abs, because it would cause the JIT compilation of opencv_cudaarithm module.

time only represents time to initialize the GpuMat and calculate its absolute value.<- show to more decimal places - don’t call it initialization delay

Understanding the importance of addressing JIT compilation delays is crucial, as this delay can have a significant impact on performance, as observed in the recorded delay of 172.30s. However, it’s worth noting that this JIT delay can be completely eliminated if desired (see Just-in-Time Compilation. By ensuring that the -DCUDA_ARCH_BIN flag is set to the compute capability of the intended devices, you can remove this delay entirely.

On the other hand, it’s essential to recognize that the delay caused by the creation of a CUDA context, which will be explored in the next section, will always be present. While this delay is generally less significant compared to JIT compilation, it is a fundamental aspect of CUDA initialization and cannot be eliminated.

CUDA context creation

In OpenCV a CUDA context is automatically created by the CUDA runtime the first time a CUDA function is executed. This is a one time delay and only takes a few seconds but its important to realize its there for example when benchmarking CUDA functions. i.e. If you want to time GpuMat creation then it is advisable perform the creation or other CUDA call once before initiating the timing. e.g.

gpuMat = cv.cuda.GpuMat(10,10,cv.CV_8U)
start = timer()
gpuMat = cv.cuda.GpuMat(10,10,cv.CV_8U)
end = timer()

As we have already initialized a CUDA context above we first need to destroy the existing CUDA context by reseting the device with cv.cuda.resetDevice() before calling time_set_device().

cv.cuda.resetDevice()
time_set_device()
Initialization time: 0.33s

As you can see the initialization delay is tiny (0.33s) compared to that from JIT compilation, however unlike the JIT compilation delay this will always be present.
